# 15. Row Level Security

## Row-Level Security (RLS) in Power BI

6

## What is Row-Level Security (RLS)?

**Row-Level Security (RLS)** is a Power BI security feature that **restricts which rows of data a user can see** in a report. Different users can open the same report but only view the data they are authorized to access.

#### Simple Definition

> **Row-Level Security (RLS) allows different users to see different rows of the same dataset based on security rules.**

For example, if a report contains sales data for all regions, RLS can ensure that:

* The **North Manager** sees only North region sales.
* The **South Manager** sees only South region sales.
* The **CEO** sees data for all regions.

***

## Why is Row-Level Security Used?

Without RLS:

All users can see all records.

With RLS:

Each user sees **only the data assigned to them**, improving security and privacy.

***

## Real-Life Example

Imagine a company has sales in four regions.

| Region | Sales      |
| ------ | ---------- |
| North  | ₹10,00,000 |
| South  | ₹8,50,000  |
| East   | ₹9,20,000  |
| West   | ₹7,80,000  |

#### Without RLS

All managers see:

* North
* South
* East
* West

#### With RLS

| User          | Visible Data |
| ------------- | ------------ |
| North Manager | North only   |
| South Manager | South only   |
| East Manager  | East only    |
| CEO           | All Regions  |

The report is the same, but the data displayed depends on the logged-in user.

***

## How Does RLS Work?

```
Power BI Dataset        │        ▼Create Security Role        │        ▼Apply DAX Filter        │        ▼Publish to Power BI Service        │        ▼Assign Users to Role        │        ▼User Opens Report        │        ▼Only Allowed Rows Are Displayed
```

***

## Types of Row-Level Security

### 1. Static Row-Level Security

Users are assigned to predefined roles.

Example:

* North Role
* South Role
* HR Role
* Finance Role

Each role has its own filter.

#### Example Filter

```
Region = "North"
```

Anyone assigned to the **North** role sees only North records.

***

### 2. Dynamic Row-Level Security

Dynamic RLS automatically filters data based on the logged-in user's email or username.

Common DAX functions include:

```
USERPRINCIPALNAME()
```

or

```
USERNAME()
```

This is ideal when many users require different access without creating separate roles for each person.

***

## Step-by-Step Procedure to Create Row-Level Security

### Step 1: Open Power BI Desktop

Open your report (`.pbix` file).

***

### Step 2: Go to the Modeling Tab

Click:

```
Modeling   ↓Manage Roles
```

The **Manage Roles** window opens.

***

### Step 3: Create a New Role

Click **Create**.

Enter a role name.

Example:

```
North Manager
```

***

### Step 4: Select the Table

Choose the table containing the column you want to filter.

Example:

```
Sales
```

***

### Step 5: Add a Filter

Suppose the table contains a **Region** column.

Enter this DAX expression:

```
[Region] = "North"
```

Click **Save**.

Now the **North Manager** role can see only rows where Region equals North.

***

### Step 6: Test the Role

Go to:

```
Modeling   ↓View As
```

Select:

```
North Manager
```

Click **OK**.

The report now displays only North region data, allowing you to verify that the security filter works.

***

### Step 7: Publish the Report

Publish the report to **Power BI Service**.

***

### Step 8: Assign Users

In Power BI Service:

1. Open the workspace.
2. Select the dataset (semantic model).
3. Choose **Security**.
4. Select the role (for example, **North Manager**).
5. Add the users or groups who belong to that role.
6. Save the changes.

When those users open the report, they will see only the data allowed by their assigned role.

***

## Example

### Sales Table

| Employee | Region | Sales     |
| -------- | ------ | --------- |
| Amit     | North  | ₹1,20,000 |
| Rahul    | South  | ₹95,000   |
| Priya    | East   | ₹1,10,000 |
| Mohan    | West   | ₹90,000   |

#### RLS Filter

```
[Region] = "North"
```

#### Visible Data

| Employee | Region | Sales     |
| -------- | ------ | --------- |
| Amit     | North  | ₹1,20,000 |

All other rows are hidden.

***

## Dynamic RLS Example

Assume you have a security table.

| Email             | Region |
| ----------------- | ------ |
| amit@company.com  | North  |
| rahul@company.com | South  |

Relationship:

```
Security Table        │        ▼Sales Table
```

Filter using:

```
[Email] = USERPRINCIPALNAME()
```

When Amit logs in, he sees North data.

When Rahul logs in, he sees South data.

The same report behaves differently for each user.

## Benefits of Row-Level Security

* Protects confidential information.
* Ensures users see only relevant data.
* Supports data privacy and governance.
* Allows one report to serve many users securely.
* Reduces the need to maintain multiple copies of the same report.
* Works seamlessly with Power BI Service.
* Simplifies report management.
* Improves compliance with organizational security policies.

***

## Advantages

| Advantage                  | Description                                             |
| -------------------------- | ------------------------------------------------------- |
| **Improved Security**      | Users can access only authorized data.                  |
| **Single Report**          | One report can be used by many departments.             |
| **Easy Maintenance**       | Update one report instead of many copies.               |
| **Better User Experience** | Each user sees only relevant information.               |
| **Scalable**               | Dynamic RLS can support hundreds or thousands of users. |
| **Privacy Compliance**     | Helps protect sensitive business information.           |

***

## Static RLS vs Dynamic RLS

| Static RLS                       | Dynamic RLS                        |
| -------------------------------- | ---------------------------------- |
| Uses fixed filters               | Uses the logged-in user's identity |
| Best for a small number of roles | Best for many users                |
| Example: `Region = "North"`      | Example: `USERPRINCIPALNAME()`     |
| Separate roles may be needed     | One role can work for many users   |

***

## Common DAX Functions Used in RLS

| Function              | Purpose                                                              |
| --------------------- | -------------------------------------------------------------------- |
| `USERPRINCIPALNAME()` | Returns the current user's email address in Power BI Service.        |
| `USERNAME()`          | Returns the current user's identity (format depends on environment). |
| `LOOKUPVALUE()`       | Retrieves matching values from another table.                        |
| `RELATED()`           | Accesses related table values.                                       |
| `IF()`                | Applies conditional logic.                                           |

***

## Best Practices

* Use **Dynamic RLS** for organizations with many users.
* Test every role using **View As** before publishing.
* Maintain a dedicated security table for user-to-role mappings.
* Assign roles to Microsoft Entra ID (Azure AD) groups where possible instead of individual users.
* Keep relationships between security tables and fact tables simple and well-designed.

***

## Common Mistakes

* Forgetting to publish the updated dataset after creating RLS.
* Not assigning users to roles in Power BI Service.
* Testing only in Desktop and not verifying in the Service.
* Applying filters to the wrong table or using incorrect relationships.
* Assuming workspace admins are restricted by RLS (they generally have broader access).
