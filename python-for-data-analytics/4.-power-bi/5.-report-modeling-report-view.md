# 5. Report Modeling (Report View)

## What is Report Modeling?

**Report Modeling in Power BI** is the process of organising and connecting data tables so that reports and dashboards can be created easily and accurately. It means **creating relationships between tables, arranging data properly, and preparing the data structure** before creating charts and reports.

<figure><img src="../../.gitbook/assets/Img.png" alt=""><figcaption></figcaption></figure>



### Why Report View is Used:

* It is the only place where visuals are actually created and arranged on a page.
* It lets you combine multiple visuals (cards, charts, maps, tables) into one dashboard story.
* It supports interactivity — clicking one visual filters the others automatically (cross-filtering).
* It allows page-by-page storytelling — for example, a Sales Dashboard page and a separate Product Dashboard page.                                        &#x20;
* It is where formatting, branding, and polish happen before a report is published or shared.



### Internal Working of Report View

Understanding what happens behind the screen helps you debug slow or broken visuals later — this is also a common interview topic.

1\.   Data model connection: Report View doesn't store data itself. It reads from the in-memory data model (built in Data/Model view) using the VertiPaq engine (Import mode) or live/DirectQuery connections.

2\.   Visual as a query: Every visual you place on the canvas (e.g., the 'Revenue by Fiscal Year' line chart in your screenshot) is really a mini DAX query. Power BI generates a DAX query behind the scenes for every visual, sends it to the model, and renders the result.

3\.   Rendering: The returned data is passed to the visual's rendering library (built-in or custom visual), which draws it as SVG/Canvas inside the browser-based rendering engine that powers Desktop's canvas.

4\.   Interactivity engine: When you click a data point (e.g., a slice of the map or a bar in the employee matrix), Power BI raises a cross-filter event that re-queries every other visual on the page bound to that field.

5\.   Refresh cycle: Any change to a filter, slicer, or field re-triggers the query-render cycle for affected visuals only — not the whole page — which is why well-designed reports feel fast.

### &#x20;1. Visualizations Pane - Full Overview

The Visualizations Pane sits in the middle column on the right side of Report View, between the Filters Pane and the Data Pane. It only becomes fully active once a visual on the canvas is selected - everything in it applies to that one selected visual.

<figure><img src="../../.gitbook/assets/Screenshot 2026-07-10 124806.png" alt=""><figcaption></figcaption></figure>

<p align="center"><em>Visualizations Pane: Build Visual tab open for a Map visual, showing the visual gallery, field wells, and Drill through toggles.</em></p>

#### 1.1 The Three Top-Level Icons

| **Icon / Tab**                        | **What it does**                                                                                                             |
| ------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------- |
| **Build Visual (grid icon)**          | Choose the visual type from the gallery, then drag fields from the Data Pane into wells like Axis, Legend, Values, Tooltips. |
| **Format Visual (paintbrush icon)**   | Style the selected visual - colors, titles, borders, data labels, backgrounds. Covered fully in Part 5.                      |
| **Analytics (magnifying-glass icon)** | Add reference lines, trend lines, forecasting, constant lines, and error bars - statistical overlays on top of the data.     |

#### 1.2 The Visual Gallery

At the top of the Build Visual tab is a grid of icons - one per visual type (Clustered Column, Line, Pie, Map, Table, Matrix, and so on, all visible in your screenshot's icon grid). Clicking an icon changes what visual the current selection renders as. Power BI tries to preserve field assignments when you swap between similar visual types (e.g., Clustered Column to Line), but wells that don't exist in the new visual type are dropped.

#### 1.3 Why Field Wells Exist

* They map raw columns and measures to specific visual roles (an axis, a color split, a size) without writing any code.
* Each well enforces sensible expectations - e.g., Values wells expect numeric/aggregatable fields, Axis wells expect categorical or date fields.
* Swapping a field in a well instantly re-queries and redraws the visual, so the effect of a choice is immediately visible.

#### 2.1 Cartesian Chart Wells (Column, Bar, Line, Area, Ribbon charts)

Your 'Revenue by Fiscal Year' line chart (from Part 1) is a Cartesian chart. Its wells are the most commonly used in all of Power BI, so start here.

#### Axis (also called Category Axis)

| **What it is**                 | The well that defines the categories or time periods spread along the horizontal axis of a chart.                                           |
| ------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------- |
| **Why it is used**             | Sets what each bar, point, or line segment represents - usually a date, a product name, or a region.                                        |
| **What data goes there**       | A dimension column: Date, Product Name, Region, Employee Name.                                                                              |
| **Recommended data type**      | Text or Date/Whole Number (categorical) - not a decimal measure.                                                                            |
| **If the wrong field is used** | Putting a numeric measure (like Revenue) here instead of a category creates a meaningless, oddly-bucketed axis instead of clean categories. |

Business example: In your Revenue by Fiscal Year chart, Axis = Fiscal Year (2013, 2014, 2015, 2016) - giving four clean points along the bottom instead of a continuous, unreadable numeric scale.

#### Legend

| **What it is**                 | The well that splits a single chart into multiple color-coded series.                                                                 |
| ------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------- |
| **Why it is used**             | Lets one chart compare several categories side-by-side instead of building a separate chart for each.                                 |
| **What data goes there**       | A second dimension column, different from the Axis field: Product Category, Region, Buying Group.                                     |
| **Recommended data type**      | Text or categorical column with a reasonably small number of distinct values (ideally under 10-15, or the legend becomes unreadable). |
| **If the wrong field is used** | Using a high-cardinality column (like Customer Name with 400+ values) explodes the legend into an unreadable rainbow.                 |

Business example: Adding 'Buying Group' to the Legend on the Revenue by Fiscal Year chart would show one colored line per buying group instead of a single blended line.

#### Values (Y Axis)

| **What it is**                 | The well that defines the numeric height/length of each bar, point, or line - the vertical axis on most charts.                 |
| ------------------------------ | ------------------------------------------------------------------------------------------------------------------------------- |
| **Why it is used**             | This is the actual number being measured and compared across categories.                                                        |
| **What data goes there**       | A measure or numeric column: Revenue, Profit, Quantity.                                                                         |
| **Recommended data type**      | Decimal, Currency, or Whole Number, ideally an aggregated measure (SUM, AVERAGE).                                               |
| **If the wrong field is used** | Dropping a text or ID column here either errors out or falls back to a meaningless COUNT, which rarely matches business intent. |

#### X Axis / Y Axis (Scatter & combo charts)

| **What it is**                 | On Scatter/Bubble charts specifically, both axes take numeric measures instead of one axis being categorical.                                                        |
| ------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Why it is used**             | Lets you plot two numeric measures against each other to find correlation or outliers - e.g., Profit vs Quantity.                                                    |
| **What data goes there**       | Two different numeric measures.                                                                                                                                      |
| **Recommended data type**      | Decimal/Whole Number for both axes.                                                                                                                                  |
| **If the wrong field is used** | Putting a categorical field on the X or Y Axis of a Scatter chart breaks the correlation view the chart is designed for - use Details/Legend for categories instead. |

Business example: Plotting Profit (Y Axis) against Quantity (X Axis) per employee would show at a glance which employees are high-volume but low-margin, versus low-volume, high-margin.

#### Secondary Values (Combo charts)

| **What it is**                 | A second Values well used only on combo visuals (Line and Clustered Column Chart), rendered against a second, independent Y-axis on the right side.   |
| ------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Why it is used**             | Lets you compare two measures on very different scales on one chart - e.g., Revenue in millions next to a Profit Margin percentage.                   |
| **What data goes there**       | A second numeric measure, usually on a different scale than the primary Values field.                                                                 |
| **Recommended data type**      | Decimal / Percentage.                                                                                                                                 |
| **If the wrong field is used** | Putting a measure on the same scale as the primary Values field wastes the secondary axis and can visually mislead by exaggerating small differences. |

Business example: Showing Revenue (columns, primary axis, in millions) alongside Profit Margin % (line, secondary axis, 0-100%) on one combo chart lets executives see both scale and efficiency together.

#### Small Multiples

| **What it is**                 | A well (available on most modern chart types) that splits one chart into a grid of many mini-charts, one per category. |
| ------------------------------ | ---------------------------------------------------------------------------------------------------------------------- |
| **Why it is used**             | Replaces a single cluttered chart with many small, comparable charts - much easier to scan than an overloaded Legend.  |
| **What data goes there**       | A dimension column with a moderate number of distinct values: Region, Product Category, Store.                         |
| **Recommended data type**      | Text / categorical.                                                                                                    |
| **If the wrong field is used** | Using a field with too many distinct values (e.g., 400 customers) produces an unreadable wall of tiny charts.          |

Business example: Splitting Revenue by Fiscal Year into Small Multiples by Region shows one mini line chart per region, all on the same scale, side-by-side - instantly comparable.

#### Tooltips

| **What it is**                 | A well that adds extra fields (or an entire custom Tooltip Page) shown in the pop-up box when a user hovers over a data point. |
| ------------------------------ | ------------------------------------------------------------------------------------------------------------------------------ |
| **Why it is used**             | Lets you show supporting context without cluttering the main chart.                                                            |
| **What data goes there**       | Any measure, or a reference to a small report page built and sized specifically as a tooltip.                                  |
| **Recommended data type**      | Numeric measures for simple tooltips; a Page reference for custom visual tooltips.                                             |
| **If the wrong field is used** | Adding too many fields makes the hover tooltip cluttered and slow to read.                                                     |

Business example: In your Map visual, Tooltips includes Profit and Quantity plus a custom Tooltip Page - hovering over any state shows a small mini-report instead of a plain text box, as seen in the Build Visual pane screenshot.



### 2.2 Map Visual Wells (Map, Filled Map, Shape Map, Azure Map)

Your 'Revenue by State Province' visual is a filled Map. Maps use a distinct set of wells built around geography instead of a simple X/Y axis.

#### Location

| **What it is**                 | The well that tells Power BI what named place to plot - city, state, country, postal code.                                                                                        |
| ------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Why it is used**             | Power BI's built-in Bing-powered geocoding turns place names into map coordinates automatically when this well is used.                                                           |
| **What data goes there**       | A text column with recognizable place names: 'State Province', 'Country', 'City'.                                                                                                 |
| **Recommended data type**      | Text (geography-recognizable).                                                                                                                                                    |
| **If the wrong field is used** | Using an ambiguous or misspelled place name (e.g., a city that exists in multiple countries without a Country field alongside it) causes points to geocode to the wrong location. |

Business example: Location = State Province lets the map auto-plot every US/Canada state without you supplying coordinates manually.

#### Latitude & Longitude

| **What it is**                 | Two wells that accept exact numeric coordinates instead of relying on name-based geocoding.                                                          |
| ------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Why it is used**             | Used when you already have precise coordinates (e.g., store GPS locations) and want to skip geocoding guesswork entirely - faster and more accurate. |
| **What data goes there**       | Decimal numeric columns holding coordinate values.                                                                                                   |
| **Recommended data type**      | Decimal Number.                                                                                                                                      |
| **If the wrong field is used** | Swapping Latitude and Longitude values, or using a non-numeric column, plots points in the ocean or off the map entirely.                            |

Business example: A retail chain with a Stores table containing exact Latitude/Longitude per branch can plot every store precisely, instead of relying on city-name geocoding that clusters everything at a city's centre point.

#### Size

| **What it is**                 | The well that controls how large each bubble/point appears on a bubble map.                                              |
| ------------------------------ | ------------------------------------------------------------------------------------------------------------------------ |
| **Why it is used**             | Lets viewers instantly compare magnitude by eye - bigger bubble means bigger number.                                     |
| **What data goes there**       | A numeric measure: Revenue, Quantity, Population.                                                                        |
| **Recommended data type**      | Decimal / Whole Number, aggregated.                                                                                      |
| **If the wrong field is used** | Using a non-additive field (like an average that shouldn't be summed further) can visually mislead when bubbles overlap. |

Business example: In the Revenue by State Province map, Size (via Values) makes higher-revenue states show as larger circles, so a viewer spots the biggest markets at a glance.

#### Color Saturation

| **What it is**                 | A well that shades each location on a gradient (light to dark) based on a numeric measure, instead of using bubble size.                            |
| ------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Why it is used**             | Useful when you want a heat-map effect - dense color = high value - especially on Filled/Shape maps where there's no 'bubble' to resize.            |
| **What data goes there**       | A numeric measure: Revenue, Profit Margin, Risk Score.                                                                                              |
| **Recommended data type**      | Decimal / Whole Number.                                                                                                                             |
| **If the wrong field is used** | Applying Color Saturation with too narrow a value range makes every region look the same shade, hiding real differences - needs a data range check. |

Business example: A Filled Map shaded by Profit Margin % lets a finance team spot low-margin states in pale colour and high-margin states in deep purple instantly.

#### Category

| **What it is**                 | A well (seen on maps and several other visuals) that assigns a distinct color per category at a location, similar to Legend. |
| ------------------------------ | ---------------------------------------------------------------------------------------------------------------------------- |
| **Why it is used**             | Splits map markers by a grouping field so multiple categories are visible on the same map, color-coded.                      |
| **What data goes there**       | A dimension column: Buying Group, Product Category.                                                                          |
| **Recommended data type**      | Text / categorical, low cardinality.                                                                                         |
| **If the wrong field is used** | Same risk as Legend - too many categories makes the color key unreadable.                                                    |

Business example: Colouring store markers by Buying Group shows at a glance which regions lean toward wholesale versus retail buyers.

#### Play Axis

| **What it is**                 | A well that adds an animated play button beneath a Scatter/Bubble chart or map, stepping through a time or sequence field.      |
| ------------------------------ | ------------------------------------------------------------------------------------------------------------------------------- |
| **Why it is used**             | Turns a static snapshot into a moving-through-time animation - powerful for showing trend and momentum in a live presentation.  |
| **What data goes there**       | A Date or sequential numeric field: Year, Quarter, Month.                                                                       |
| **Recommended data type**      | Date/Whole Number, ordered.                                                                                                     |
| **If the wrong field is used** | Using a non-sequential or non-unique-order field makes the animation jump around meaninglessly instead of progressing smoothly. |

Business example: Adding Fiscal Year to Play Axis on a Profit-vs-Quantity bubble chart animates the bubbles moving year by year - a strong visual for a leadership presentation on employee performance trends.















































































































































### Filters Pane

Definition: The Filters Pane is the column on the right side of the screen (leftmost of the three right-hand panes) that lets you restrict which data appears in one visual, one page, or the entire report.

Screenshot reference: Your Filters Pane shows fields like Employee, Profit, Quantity, Revenue (all set to "is (All)") under Filters on this Visual for the matrix chart, and a Customer filter set to "is not (Blank)" under Filters on this page.

#### The three filter scopes visible in your screenshots

| **Scope**              | **Effect**                                                                                                        |
| ---------------------- | ----------------------------------------------------------------------------------------------------------------- |
| Filters on this visual | Applies only to the single selected visual (e.g., the Employee matrix). Other visuals on the page are unaffected. |
| Filters on this page   | Applies to every visual on the current page only (e.g., Customer is not Blank, applied page-wide).                |
| Filters on all pages   | Applies to every visual on every page of the report — used for global rules like excluding test data.             |

<figure><img src="../../.gitbook/assets/Screenshot 2026-07-10 124806.png" alt=""><figcaption></figcaption></figure>

#### Why it exists

* Lets report builders remove irrelevant or bad data (e.g., blank customer names) without touching the data model itself.
* Enables layered control — a broad rule at the report level, a narrower rule at the page level, and a precise rule at the visual level.
* Powers drill-through — drill-through filters are a special filter type triggered by right-click actions.

#### Advantages

* No DAX or coding needed — filters are point-and-click.
* Changes are visible instantly on the canvas.
* Filters can be locked to prevent report viewers from accidentally changing them.

#### Disadvantages / Watch-outs

* Overusing visual-level filters instead of a shared slicer can make dashboards inconsistent across visuals.
* Hidden filters (used for calculations behind the scenes) can confuse other report editors if not documented.

#### Real-life Business Example

In your dashboard, "Customer is not (Blank)" on Filters on this page ensures the customer table only shows real, named customers — hiding transactions where the customer field wasn't captured, which would otherwise create a messy blank row at the top of every table and chart on that page.
