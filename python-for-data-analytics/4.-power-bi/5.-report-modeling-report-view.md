# 5. Report Modeling (Report View)

## What is Report Modeling?

**Report Modeling in Power BI** is the process of organising and connecting data tables so that reports and dashboards can be created easily and accurately. It means **creating relationships between tables, arranging data properly, and preparing the data structure** before creating charts and reports.

<figure><img src="../../.gitbook/assets/Img.png" alt=""><figcaption></figcaption></figure>



### Why Report View is Used:

* It is the only place where visuals are actually created and arranged on a page.
* It lets you combine multiple visuals (cards, charts, maps, tables) into one dashboard story.
* It supports interactivity — clicking one visual filters the others automatically (cross-filtering).
* It allows page-by-page storytelling — for example, a Sales Dashboard page and a separate Product Dashboard page.                                        &#x20;
* It is where formatting, branding, and polish happen before a report is published or shared.



### Internal Working of Report View

Understanding what happens behind the screen helps you debug slow or broken visuals later — this is also a common interview topic.

1\.   Data model connection: Report View doesn't store data itself. It reads from the in-memory data model (built in Data/Model view) using the VertiPaq engine (Import mode) or live/DirectQuery connections.

2\.   Visual as a query: Every visual you place on the canvas (e.g., the 'Revenue by Fiscal Year' line chart in your screenshot) is really a mini DAX query. Power BI generates a DAX query behind the scenes for every visual, sends it to the model, and renders the result.

3\.   Rendering: The returned data is passed to the visual's rendering library (built-in or custom visual), which draws it as SVG/Canvas inside the browser-based rendering engine that powers Desktop's canvas.

4\.   Interactivity engine: When you click a data point (e.g., a slice of the map or a bar in the employee matrix), Power BI raises a cross-filter event that re-queries every other visual on the page bound to that field.

5\.   Refresh cycle: Any change to a filter, slicer, or field re-triggers the query-render cycle for affected visuals only — not the whole page — which is why well-designed reports feel fast.

### &#x20;1. Visualizations Pane - Full Overview

The Visualizations Pane sits in the middle column on the right side of Report View, between the Filters Pane and the Data Pane. It only becomes fully active once a visual on the canvas is selected - everything in it applies to that one selected visual.

<figure><img src="../../.gitbook/assets/Screenshot 2026-07-10 124806.png" alt=""><figcaption></figcaption></figure>

<p align="center"><em>Fig 1 - Visualizations Pane: Build Visual tab open for a Map visual, showing the visual gallery, field wells, and Drill through toggles.</em></p>

### 1.1 The Three Top-Level Icons

| **Icon / Tab**                    | **What it does**                                                                                                             |
| --------------------------------- | ---------------------------------------------------------------------------------------------------------------------------- |
| Build Visual (grid icon)          | Choose the visual type from the gallery, then drag fields from the Data Pane into wells like Axis, Legend, Values, Tooltips. |
| Format Visual (paintbrush icon)   | Style the selected visual - colors, titles, borders, data labels, backgrounds. Covered fully in Part 5.                      |
| Analytics (magnifying-glass icon) | Add reference lines, trend lines, forecasting, constant lines, and error bars - statistical overlays on top of the data.     |

### 1.2 The Visual Gallery

At the top of the Build Visual tab is a grid of icons - one per visual type (Clustered Column, Line, Pie, Map, Table, Matrix, and so on, all visible in your screenshot's icon grid). Clicking an icon changes what visual the current selection renders as. Power BI tries to preserve field assignments when you swap between similar visual types (e.g., Clustered Column to Line), but wells that don't exist in the new visual type are dropped.

### 1.3 Why Field Wells Exist

* They map raw columns and measures to specific visual roles (an axis, a color split, a size) without writing any code.
* Each well enforces sensible expectations - e.g., Values wells expect numeric/aggregatable fields, Axis wells expect categorical or date fields.
* Swapping a field in a well instantly re-queries and redraws the visual, so the effect of a choice is immediately visible.

### Filters Pane

Definition: The Filters Pane is the column on the right side of the screen (leftmost of the three right-hand panes) that lets you restrict which data appears in one visual, one page, or the entire report.

Screenshot reference: Your Filters Pane shows fields like Employee, Profit, Quantity, Revenue (all set to "is (All)") under Filters on this Visual for the matrix chart, and a Customer filter set to "is not (Blank)" under Filters on this page.

#### The three filter scopes visible in your screenshots

| **Scope**              | **Effect**                                                                                                        |
| ---------------------- | ----------------------------------------------------------------------------------------------------------------- |
| Filters on this visual | Applies only to the single selected visual (e.g., the Employee matrix). Other visuals on the page are unaffected. |
| Filters on this page   | Applies to every visual on the current page only (e.g., Customer is not Blank, applied page-wide).                |
| Filters on all pages   | Applies to every visual on every page of the report — used for global rules like excluding test data.             |

<figure><img src="../../.gitbook/assets/Screenshot 2026-07-10 124806.png" alt=""><figcaption></figcaption></figure>

#### Why it exists

* Lets report builders remove irrelevant or bad data (e.g., blank customer names) without touching the data model itself.
* Enables layered control — a broad rule at the report level, a narrower rule at the page level, and a precise rule at the visual level.
* Powers drill-through — drill-through filters are a special filter type triggered by right-click actions.

#### Advantages

* No DAX or coding needed — filters are point-and-click.
* Changes are visible instantly on the canvas.
* Filters can be locked to prevent report viewers from accidentally changing them.

#### Disadvantages / Watch-outs

* Overusing visual-level filters instead of a shared slicer can make dashboards inconsistent across visuals.
* Hidden filters (used for calculations behind the scenes) can confuse other report editors if not documented.

#### Real-life Business Example

In your dashboard, "Customer is not (Blank)" on Filters on this page ensures the customer table only shows real, named customers — hiding transactions where the customer field wasn't captured, which would otherwise create a messy blank row at the top of every table and chart on that page.
